# Next.js TypeScript Project Setup Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Bootstrap a production-ready Next.js 14+ full-stack project with TypeScript, Tailwind CSS, Prisma/PostgreSQL, and Vitest.

**Architecture:** App Router with React Server Components; all database access is server-side only via Server Actions or Route Handlers. The `src/lib/db.ts` Prisma singleton is the single DB entry point — it is never imported from client components.

**Tech Stack:** Next.js 14+ (App Router), TypeScript (strict), Tailwind CSS, Prisma ORM, PostgreSQL, Vitest, React Testing Library, Prettier + prettier-plugin-tailwindcss, ESLint (eslint-config-next)

## Task 2: Verify TypeScript strict mode

**Files:**

- Modify: `tsconfig.json` (verify/set strict flags)

- [ ] **Step 1: Open `tsconfig.json` and confirm `strict: true` is set**

The `compilerOptions` block must contain:

```json
{
  "compilerOptions": {
    "strict": true,
    "noUncheckedIndexedAccess": true
  }
}
```

`noUncheckedIndexedAccess` makes array/object index access return `T | undefined`, catching a large class of runtime bugs. Add it if missing.

- [ ] **Step 2: Run the type checker**

import { describe, it, expect } from 'vitest'
import { cn } from '@/lib/utils'

describe('cn', () => {
  it('concatenates plain class names', () => {
    expect(cn('foo', 'bar')).toBe('foo bar')
  })

  it('drops duplicate Tailwind classes, keeping the last one', () => {
    // tailwind-merge: p-4 is overridden by p-2

  })

  it('accepts arrays and objects (clsx passthrough)', () => {

- [ ] **Step 6: Write the failing test for `db.ts`**

Create `src/lib/db.test.ts`:

```ts
import { describe, it, expect, vi } from 'vitest'

// Mock PrismaClient before importing db — prevents real DB connection in tests
vi.mock('@prisma/client', () => {
  const PrismaClient = vi.fn().mockImplementation(() => ({
    $connect: vi.fn(),
    $disconnect: vi.fn(),
  }))
  return { PrismaClient }
})

describe('db', () => {
  it('exports a prisma singleton', async () => {
    const { prisma } = await import('@/lib/db')
    expect(prisma).toBeDefined()
    expect(typeof prisma).toBe('object')
  })

  it('reuses the same instance across imports (singleton)', async () => {
    const { prisma: a } = await import('@/lib/db')
    const { prisma: b } = await import('@/lib/db')
    expect(a).toBe(b)
  })
})
```

- [ ] **Step 7: Run the test to confirm it fails**

```bash
npm test
```

Expected: FAIL — `Cannot find module '@/lib/db'`

- [ ] **Step 8: Create `src/lib/db.ts`**

```ts
import { PrismaClient } from '@prisma/client'

const globalForPrisma = globalThis as unknown as {
  prisma: PrismaClient | undefined
}

export const prisma = globalForPrisma.prisma ?? new PrismaClient()

if (process.env.NODE_ENV !== 'production') {
  globalForPrisma.prisma = prisma
}
```

> **Why the global cache?** Next.js hot-reloads modules in development, which would create a new PrismaClient on every reload — exhausting the DB connection pool. Caching on `globalThis` (which persists across HMR) avoids this.
